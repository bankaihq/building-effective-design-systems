## Who uses Design Systems? - Designers - Developers - Product Owners - Copy Writers - Data Scientists - Marketeers Notes: - Designers (documenting design choices, toolbox) - Developers (toolbox, ref. for how to use thing) - Product Owners (Getting a sense of how time is invested) - Copy Writers (providing context for UI texts) - Data Scientists (how to map data into components) - Marketeers (docs about brand identity) Single Source of thruth for an organisation. onboarding new people
## How to build something that can be used by everyone? Key parts: 1. Work __together__ 2. Keep __structure__ 3. Grow __safely__ Notes: 1. Work __together__; through __co-ownership__ 2. Keep structure; making it __accessible__ and __maintainable__ 3. grow __safely__.
## Part 1: Work together In order for it to be effective the system needs to be owned __together__. The best way to achieve this is to work on it __together__. Notes: Solves designer vs developer problem. This is the only way it will be become a single source of truth.
### A common language Everyone uses __different terms__ for the __same things__. In order to understand __each other__ you need to speak the __same language__.

### Composing components, not building them Notes: I call it composing component, because its a process. We are not a mindless machines. Or at least, you shouldn't be. Thinking is like, good.
### Making components is a __creative process__. Yes, you're an artist now. Notes: > So next time when your parents ask you "so, what do you do again?" > You say, "I am an artist, a digital artist"
### Composing components is done through __collaboration__ - Discuss the design with the person who made it. - Does it contain all the info we need? - We ask critical questions on how it should work. - Are animations and transitions defined. If not, should they? Notes: Have a discussion about, using clear arguments and explain them.

## Part 3: Grow __safely__ Notes: This part is all about quality, what methods should you use?
### Each thing is documented The bigger a team gets the harder it is to communicate. Document once and you will never need to explain it again. Documentation tools: - __StyleGuidist__ ([React](https://react-styleguidist.js.org/) or [Vue](https://vue-styleguidist.github.io/)) ‚ù§Ô∏è - [StoryBook](https://storybook.js.org/) works for Vue, React, Svelte and others. [An example](http://localhost:6060/#/Organisms/DetailedInfoHeader). Notes: - Foundation Tokens - components - utilities
### Each functionality needs to have tests - Start with manual testing scripts - Translate those to code-based test - Automate your tests
### Don't re-invent the wheel. Use methods that are well thought out and established, like: - Structured version control (GIT) - Linting (predefined rules) - __Code reviews__ through __Pull/Merge Request__ - Integration and end-to-end testing - Automate this all! Notes: - for structured version control: [__git flow__](https://nvie.com/posts/a-successful-git-branching-model/) or [__github flow__](https://guides.github.com/introduction/flow/) - add linting (like, eslint standard or AirBNB). Stop wasting time having tabs-vs-spaces discussions or wether or not something needs an extra comma behind it. - do __code reviews__ through __Pull/Merge Request__ - set up integration testing and end-to-end testing - automate the hell out of it! Make it your mantra. (maybe later, tweak it to fit your needs.)
### So summarised: - Communicate, start talking in easy terms and be nice to people. - Don't re-invent the wheel

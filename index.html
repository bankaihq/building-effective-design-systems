## How to build something that can be used by everyone? Key parts: 1. Work __together__ 2. Keep __structure__ 3. Scale __safely__ Notes: 1. Work __together__; through __co-ownership__ 2. Keep structure; making it __accessible__ and __maintainable__ 3. Scale __safely__.

## Structure: the most basic values We use __Foundation Tokens__ to define the most __basic values__ of the design's __inherit__ logic. - __Colours__, and how they are, used. - __Typography__, headings labels, etc. - __Sizings__, spaces between, and sizes of components
Pallette colours: ```scss $colour-primary: #E64D1F; // soft-red $colour-primary-high-contrast: #A41111; // dark-red $colour-secondary: #036F7B; // grey-blue $colour-secondary-high-contrast: #174A5B; // dark-grey-blue ``` Pallette colours applied to base values: ```scss // For headings and labels: $colour-main-item: $colour-primary; $colour-main-item-border: $colour-primary-high-contrast; // For Call to Action items, like buttons. $colour-cta: $colour-primary; $colour-cta-border: $colour-primary-high-contrast; // For active elements, like buttons, but that need to // draw less attention, like table options. $colour-support: $colour-secondary; $colour-support-border: $colour-secondary-high-contrast; ```

## Part 3: Scale __safely__ Notes: This part is all about quality, what methods should you use?
### Scale safely: Each thing is documented The bigger a team gets the harder it is to communicate. Document once and you will never need to explain it again. Documentation tools: - __StyleGuidist__ ([React](https://react-styleguidist.js.org/) or [Vue](https://vue-styleguidist.github.io/)) ‚ù§Ô∏è - [StoryBook](https://storybook.js.org/) works for Vue, React, Svelte and others. [An example](http://localhost:6060/#/Organisms/DetailedInfoHeader). Notes: - Foundation Tokens - components - utilities
### Each functionality needs to have tests - Start with manual testing scripts - Translate those to code-based test - Automate your tests
### Don't re-invent the wheel. Use methods that are well thought out and established, like: - Structured version control (GIT + [__git flow__](https://nvie.com/posts/a-successful-git-branching-model/)) - Linting (predefined rules, like, eslint standard or AirBNB) - __Code reviews__ through __Pull/Merge Request__ - Integration and end-to-end testing (We use cypress, for everything) - All automated through CI/CD and git hooks Notes: - for structured version control: [__git flow__](https://nvie.com/posts/a-successful-git-branching-model/) or [__github flow__](https://guides.github.com/introduction/flow/) - add linting (like, eslint standard or AirBNB). Stop wasting time having tabs-vs-spaces discussions or wether or not something needs an extra comma behind it. - do __code reviews__ through __Pull/Merge Request__ - set up integration testing and end-to-end testing - automate the hell out of it!
